Guida d'uso

Telemetria live per i tuoi task asincroni

task-telemetry è una piccola libreria Java che permette a un processo lungo o asincrono (un import, un upload, un job schedulato…) di comunicare in tempo reale a che punto è, se è ancora vivo e come è finito — verso uno o più osservatori.

Il task non sa chi lo ascolta: emette eventi e basta. Un osservatore si registra e li riceve. Questa guida ti mostra, passo passo, tutto quello che puoi fare — senza dover capire come la libreria funziona dentro.

In due righe: il task apre un TaskReporter e chiama progress(...), info(...), completed(...). Chi osserva si registra con listen() o, più comodo, con TaskWatcher.

Concetti base

  • Task: il tipo di lavoro, identificato da un nome (es. "IMPORT_CLIENTI").
  • Execution: una singola esecuzione di quel task, con un executionId generato automaticamente.
  • correlationKey (facoltativa): un riferimento al tuo dominio applicativo da legare all'esecuzione (es. il nome del file, l'id della pratica).
  • Evento (TaskEvent): ciò che il task emette — inizio, avanzamento, messaggi, battito, fine.
  • Reporter (TaskReporter): l'oggetto con cui il task emette gli eventi.
  • Listener: la funzione che riceve gli eventi lato osservatore.
  • Transport: il "tubo" su cui viaggiano gli eventi (in memoria nella stessa JVM, oppure via socket tra processi).
  • Runtime (TaskTelemetry): il punto d'ingresso che crea i reporter e registra gli osservatori.

Installazione

Richiede Java 17. Aggiungi la dipendenza Maven:

pom.xml
<dependency>
    <groupId>org.tasktelemetry</groupId>
    <artifactId>task-telemetry</artifactId>
    <version>1.0-SNAPSHOT</version>
</dependency>

Il core non ha dipendenze a runtime (solo la libreria standard di Java).

Esempio minimo

Nello stesso programma: creiamo il runtime, ci mettiamo in ascolto, eseguiamo un task.

HelloTelemetry.java
try (TaskTelemetry telemetry = TaskTelemetry.defaults()) {

    // 1) chi osserva si registra
    telemetry.listen()
        .taskName("SALUTO")
        .onEvent(event -> System.out.println(event.type() + " " + event.message()))
        .start();

    // 2) il task emette i suoi eventi
    try (TaskReporter reporter = telemetry.start("SALUTO")) {
        reporter.progress(0, "inizio");
        reporter.progress(100, "fatto");
        reporter.completed("finito");
    }
}

TaskTelemetry.defaults() crea un runtime pronto all'uso (transport in memoria, battito automatico ogni 5 secondi). Sia il produttore sia l'osservatore usano la stessa istanza telemetry — questo è ciò che li collega. (Più avanti vediamo come collegarli quando sono componenti o processi separati.)

Emettere eventi (il task)

Il task ottiene un TaskReporter da telemetry.start(...) e lo usa con try-with-resources. All'apertura viene emesso automaticamente l'evento STARTED; alla chiusura, se non hai già concluso, viene emesso CANCELLED.

Produttore
// "IMPORT_CLIENTI" = nome del task; "file.csv" = correlationKey (facoltativa)
try (TaskReporter reporter = telemetry.start("IMPORT_CLIENTI", "file.csv")) {

    reporter.progress(0, "Avvio import");   // percentuale 0–100 + messaggio
    reporter.info("Lettura del file");          // messaggio informativo
    reporter.warning("Riga 42 ignorata");       // avviso non bloccante
    reporter.progress(50, "Metà fatto");
    reporter.progress(100, "Completato");

    reporter.completed("Import riuscito");       // fine con successo
}

Metodi disponibili sul reporter:

  • progress(int percentuale, String messaggio) — avanzamento 0–100.
  • info(String) / warning(String) — messaggi.
  • completed(String) — fine con successo.
  • failed(Throwable) — fine con errore.
  • cancelled(String) — interruzione volontaria.
Il task non sa chi ascolta. Emetti e basta: se nessuno è in ascolto, gli eventi semplicemente non vengono consegnati (consegna "best-effort", nessuna coda, nessuno storico).

Il battito automatico (heartbeat)

Finché il reporter è aperto e il task tace (non emette nulla), la libreria manda da sola un evento HEARTBEAT a intervalli regolari: serve a dire "sono ancora vivo". Non devi fare nulla — non chiami tu il battito. Gli eventi normali (un progress, un info) valgono già come segnale di vita e rimandano il battito.

L'intervallo si configura sul runtime (default 5 secondi):

Battito ogni 2 secondi
TaskTelemetry telemetry = TaskTelemetry.builder()
        .heartbeatInterval(Duration.ofSeconds(2))
        .build();

Concludere: completato, fallito, annullato

Un task finisce sempre con un evento terminale. Dopo un terminale non puoi più emettere.

Successo o errore
try (TaskReporter reporter = telemetry.start("EXPORT")) {
    eseguiLavoro(reporter);
    reporter.completed("Export riuscito");
} catch (Exception e) {
    // in caso di errore lo comunichi così:
    // (nota: dopo un terminale il reporter non emette più)
}

Con failed(eccezione) l'evento FAILED porta come message il testo di eccezione.toString() (tipo dell'eccezione + messaggio). Non porta un oggetto strutturato né lo stack trace: così il dato arriva identico sia nella stessa JVM sia tra processi.

Chiusura senza terminale. Se chiudi il reporter (es. fine del try-with-resources) senza aver chiamato completed/failed, per default viene emesso CANCELLED. Lo puoi cambiare (vedi Configurazione).

Ascoltare gli eventi (l'osservatore)

Ti registri con listen(), scegli cosa ascoltare e fornisci una funzione che riceve ogni evento:

Consumatore base
ListenerHandle handle = telemetry.listen()
        .taskName("IMPORT_CLIENTI")
        .onEvent(event -> {
            System.out.println(event.type() + " #" + event.sequenceNumber());
            if (event.progress() != null) {
                aggiornaBarra(event.progress());
            }
        })
        .start();

// quando non ti serve più:
handle.stop();

Ogni evento è un TaskEvent con: taskName, executionId, correlationKey, type, timestamp, sequenceNumber (numero crescente per rilevare buchi), e i campi opzionali message e progress.

Filtrare gli eventi

Puoi restringere ciò che ricevi. I filtri si combinano in AND; quelli non impostati accettano tutto.

Filtri
telemetry.listen()
        .taskName("IMPORT_CLIENTI")
        .correlationKey("file.csv")
        .eventType(TaskEventType.COMPLETED)   // solo gli eventi di fine
        .onEvent(event -> ...)
        .start();

Filtri disponibili: taskName, executionId, correlationKey, eventType.

TaskWatcher: il modo comodo per seguire un task

Se il tuo obiettivo è "aspetta che il task sia in corso, mostrami l'avanzamento, dimmi quando finisce", TaskWatcher fa tutto per te: niente listener, niente gestione dello stato a mano.

Consumatore con TaskWatcher
try (TaskWatcher watcher = new TaskWatcher(transport, "IMPORT_CLIENTI")) {

    watcher.onProgress(percento -> aggiornaBarra(percento));
    watcher.onHeartbeat(() -> log("il task è ancora vivo"));

    // aspetta fino a 5s che il task sia in corso
    if (!watcher.awaitStart(Duration.ofSeconds(5))) {
        System.out.println("Nessun task in corso");
        return;
    }

    // blocca finché finisce (o finché lo si dà per perso)
    TaskExecutionStatus esito = watcher.awaitCompletion();
    System.out.println("Esito: " + esito); // COMPLETED / FAILED / CANCELLED / LOST
}
  • onProgress(percento) — chiamata a ogni evento di avanzamento.
  • onHeartbeat() — chiamata a ogni battito (separata dai progress).
  • awaitStart(timeout) — true se il task è in corso, false se non compare entro il tempo.
  • awaitCompletion() — blocca fino alla fine e ritorna lo stato finale.
La liveness è automatica. Non devi controllare tu i battiti: se il task muore (smette di emettere), awaitCompletion() ritorna LOST invece di restare bloccato per sempre.

Lo stato di vita del task

Dal flusso di eventi la libreria ricava uno stato osservabile lato consumatore:

RUNNING — riceve eventi, è vivo STALE — silenzio oltre la soglia LOST — silenzio troppo lungo, probabilmente morto COMPLETED / FAILED / CANCELLED — finito

Con TaskWatcher questo è già gestito (lo ottieni da awaitCompletion()). Se ti serve interrogarlo direttamente puoi usare TaskHeartbeatMonitor, configurando le soglie staleAfter e lostAfter.

Soglie di vita
// stale dopo 15s di silenzio, lost dopo 60s
new TaskWatcher(transport, "IMPORT_CLIENTI",
        Duration.ofSeconds(15), Duration.ofSeconds(60));

Stessa JVM oppure processi separati

La cosa da capire: produttore e consumatore comunicano solo se condividono lo stesso transport. Non basta che usino entrambi task-telemetry.

Nella stessa JVM

Condividi la stessa istanza di TaskTelemetry (o lo stesso transport) tra le due parti, come nell'esempio minimo.

Tra processi diversi (stessa macchina)

Modello a 2 attori, senza intermediari: il task fa da server (apre una porta ed emette verso chi è connesso) e il client si connette e riceve gli eventi dal momento della connessione in poi.

Processo task (server)
try (SocketServerTaskTransport transport = new SocketServerTaskTransport(5555);
     TaskTelemetry telemetry = TaskTelemetry.builder()
             .transport(transport)
             .heartbeatInterval(Duration.ofSeconds(2))
             .build()) {

    try (TaskReporter reporter = telemetry.start("UPLOAD", "foto.zip")) {
        reporter.progress(0, "inizio");
        // ... lavoro ...
        reporter.completed("fatto");
    }
}
Processo client
try (SocketClientTaskTransport transport =
             new SocketClientTaskTransport("localhost", 5555);
     TaskWatcher watcher = new TaskWatcher(transport, "UPLOAD")) {

    watcher.onProgress(p -> System.out.println("upload al " + p + "%"));
    watcher.awaitStart(Duration.ofSeconds(30));
    System.out.println(watcher.awaitCompletion());
}
Attenzione all'ordine: con questo modello il client si connette a un task già avviato. Se parte prima che il task abbia aperto la porta, la connessione fallisce con TaskUnreachableException (gestiscila a livello applicativo: riprova o segnala). Inoltre gli eventi emessi prima della connessione non vengono recuperati.

Gestione errori di consegna

Se la pubblicazione di un evento fallisce (es. un osservatore che lancia un'eccezione), per default il task non si rompe: l'errore viene loggato e il lavoro prosegue. Puoi scegliere la politica:

Politiche di errore
TaskTelemetry.builder()
    .errorHandler(TaskTelemetryErrorHandler.ignore())   // silenzioso
    // .errorHandler(TaskTelemetryErrorHandler.logging())  // default: logga
    // .errorHandler(TaskTelemetryErrorHandler.rethrow())  // rilancia (fa fallire il task)
    .build();

Tipi di evento e dati trasportati

TipoQuandoDati utili
STARTEDAll'apertura del reporter—
PROGRESSprogress(...)progress (0–100) + message
INFOinfo(...)message
WARNINGwarning(...)message
HEARTBEATAutomatico, durante il silenzio—
COMPLETEDcompleted(...)message
FAILEDfailed(...)message (Throwable.toString())
CANCELLEDcancelled(...) o chiusuramessage
Ogni evento porta solo dati scalari più message e progress: viaggiano identici nella stessa JVM e tra processi diversi. Per inviare dati strutturati a chi osserva, mettili nel message (es. una stringa JSON che il consumatore interpreta).

Cosa NON fa (di proposito)

Per restare semplice e prevedibile, la libreria non:

  • esegue o schedula i task (li osserva soltanto);
  • garantisce la consegna o conserva uno storico (è best-effort, live);
  • permette comandi dal consumatore al task (la comunicazione è solo in uscita);
  • è un message broker o una libreria reattiva.

È tutto qui. Apri un reporter, emetti eventi, osservali. Buon lavoro.

User guide

Live telemetry for your asynchronous tasks

task-telemetry is a small Java library that lets a long-running or asynchronous process (an import, an upload, a scheduled job…) report live progress, liveness and final outcome to one or more observers.

The task does not know who is listening: it just emits events. An observer registers and receives them. This guide shows, step by step, what you can do without having to understand the internals of the library.

In two lines: the task opens a TaskReporter and calls progress(...), info(...), completed(...). The observer registers with listen() or, more conveniently, with TaskWatcher.

Core concepts

  • Task: the kind of work, identified by a name (for example "IMPORT_CLIENTS").
  • Execution: one concrete run of that task, with an automatically generated executionId.
  • correlationKey (optional): an application-domain reference attached to the execution (for example a file name or business id).
  • Event (TaskEvent): what the task emits — start, progress, messages, heartbeat, completion.
  • Reporter (TaskReporter): the object used by the task to emit events.
  • Listener: the function that receives events on the observer side.
  • Transport: the "pipe" used by events (in memory in the same JVM, or via socket between processes).
  • Runtime (TaskTelemetry): the entry point used to create reporters and register observers.

Installation

Requires Java 17. Add the Maven dependency:

pom.xml
<dependency>
    <groupId>org.tasktelemetry</groupId>
    <artifactId>task-telemetry</artifactId>
    <version>1.0-SNAPSHOT</version>
</dependency>

The core has no runtime dependencies (only the Java standard library).

Minimal example

In the same program: create the runtime, register a listener, run a task.

HelloTelemetry.java
try (TaskTelemetry telemetry = TaskTelemetry.defaults()) {

    // 1) the observer registers
    telemetry.listen()
        .taskName("GREETING")
        .onEvent(event -> System.out.println(event.type() + " " + event.message()))
        .start();

    // 2) the task emits its events
    try (TaskReporter reporter = telemetry.start("GREETING")) {
        reporter.progress(0, "started");
        reporter.progress(100, "done");
        reporter.completed("completed");
    }
}

TaskTelemetry.defaults() creates a ready-to-use runtime (in-memory transport, automatic heartbeat every 5 seconds). Both producer and observer use the same telemetry instance — that is what connects them. Later, we will see how to connect them when they are separate components or processes.

Emitting events (the task)

The task obtains a TaskReporter from telemetry.start(...) and uses it with try-with-resources. Opening the reporter automatically emits a STARTED event; closing it without a terminal event emits CANCELLED by default.

Producer
// "IMPORT_CLIENTS" = task name; "file.csv" = optional correlationKey
try (TaskReporter reporter = telemetry.start("IMPORT_CLIENTS", "file.csv")) {

    reporter.progress(0, "Import started");   // percentage 0–100 + message
    reporter.info("Reading file");              // informational message
    reporter.warning("Line 42 skipped");        // non-blocking warning
    reporter.progress(50, "Halfway done");
    reporter.progress(100, "Completed");

    reporter.completed("Import successful");      // successful terminal event
}

Available reporter methods:

  • progress(int percent, String message) — progress from 0 to 100.
  • info(String) / warning(String) — messages.
  • completed(String) — successful completion.
  • failed(Throwable) — completion with error.
  • cancelled(String) — voluntary interruption.
The task does not know who is listening. Emit and move on: if nobody is listening, events are simply not delivered (best-effort delivery, no queue, no history).

Automatic heartbeat

While the reporter is open and the task is silent (it emits nothing), the library automatically sends a HEARTBEAT event at regular intervals. It means "I am still alive". You do not call it manually. Normal events (progress, info, and so on) already count as liveness signals and postpone the next heartbeat.

The interval is configured on the runtime (default: 5 seconds):

Heartbeat every 2 seconds
TaskTelemetry telemetry = TaskTelemetry.builder()
        .heartbeatInterval(Duration.ofSeconds(2))
        .build();

Ending: completed, failed, cancelled

A task always ends with a terminal event. After a terminal event, the reporter cannot emit anymore.

Success or failure
try (TaskReporter reporter = telemetry.start("EXPORT")) {
    runWork(reporter);
    reporter.completed("Export successful");
} catch (Exception e) {
    // communicate the error with reporter.failed(e)
    // when the reporter is still available in your control flow
}

With failed(exception), the FAILED event carries exception.toString() as its message (exception type + message). It does not carry a structured object or stack trace, so the data travels the same way in-memory and across processes.

Closing without a terminal event. If the reporter is closed (for example at the end of try-with-resources) without completed/failed, CANCELLED is emitted by default. You can change that behavior in Configuration.

Listening to events (the observer)

Register with listen(), choose what to listen to and provide a function that receives each event:

Basic consumer
ListenerHandle handle = telemetry.listen()
        .taskName("IMPORT_CLIENTS")
        .onEvent(event -> {
            System.out.println(event.type() + " #" + event.sequenceNumber());
            if (event.progress() != null) {
                updateProgressBar(event.progress());
            }
        })
        .start();

// when you no longer need it:
handle.stop();

Each event is a TaskEvent with: taskName, executionId, correlationKey, type, timestamp, sequenceNumber (monotonic number useful to detect gaps), and optional message and progress.

Filtering events

You can restrict what you receive. Filters are combined with AND; unset filters match everything.

Filters
telemetry.listen()
        .taskName("IMPORT_CLIENTS")
        .correlationKey("file.csv")
        .eventType(TaskEventType.COMPLETED)   // only terminal success events
        .onEvent(event -> ...)
        .start();

Available filters: taskName, executionId, correlationKey, eventType.

TaskWatcher: the convenient way to follow a task

If your goal is "wait until the task is running, show me progress, tell me when it ends", TaskWatcher does that for you: no manual listener plumbing, no manual status handling.

Consumer with TaskWatcher
try (TaskWatcher watcher = new TaskWatcher(transport, "IMPORT_CLIENTS")) {

    watcher.onProgress(percent -> updateProgressBar(percent));
    watcher.onHeartbeat(() -> log("task is still alive"));

    // wait up to 5s for the task to be running
    if (!watcher.awaitStart(Duration.ofSeconds(5))) {
        System.out.println("No running task");
        return;
    }

    // blocks until it ends (or until it is considered lost)
    TaskExecutionStatus outcome = watcher.awaitCompletion();
    System.out.println("Outcome: " + outcome); // COMPLETED / FAILED / CANCELLED / LOST
}
  • onProgress(percent) — called for every progress event.
  • onHeartbeat() — called for every heartbeat (separate from progress).
  • awaitStart(timeout) — true if the task is running, false if it does not appear in time.
  • awaitCompletion() — blocks until completion and returns the final status.
Liveness is automatic. You do not check heartbeats manually: if the task dies and stops emitting, awaitCompletion() returns LOST instead of blocking forever.

Task liveness status

The library derives an observable consumer-side status from the event stream:

RUNNING — receiving events, alive STALE — silence beyond the threshold LOST — too much silence, probably dead COMPLETED / FAILED / CANCELLED — finished

With TaskWatcher, this is already handled (you get it from awaitCompletion()). If you need to inspect it directly, use TaskHeartbeatMonitor, configuring staleAfter and lostAfter.

Liveness thresholds
// stale after 15s of silence, lost after 60s
new TaskWatcher(transport, "IMPORT_CLIENTS",
        Duration.ofSeconds(15), Duration.ofSeconds(60));

Same JVM or separate processes

The key point: producer and consumer communicate only if they share the same transport. It is not enough for both of them to use task-telemetry.

Same JVM

Share the same TaskTelemetry instance (or the same transport) between producer and consumer, as in the minimal example.

Separate processes (same machine)

Two-actor model, no broker: the task acts as the server (opens a port and emits to connected clients), while the client connects and receives events from that moment on.

Task process (server)
try (SocketServerTaskTransport transport = new SocketServerTaskTransport(5555);
     TaskTelemetry telemetry = TaskTelemetry.builder()
             .transport(transport)
             .heartbeatInterval(Duration.ofSeconds(2))
             .build()) {

    try (TaskReporter reporter = telemetry.start("UPLOAD", "photo.zip")) {
        reporter.progress(0, "started");
        // ... real work ...
        reporter.completed("done");
    }
}
Client process
try (SocketClientTaskTransport transport =
             new SocketClientTaskTransport("localhost", 5555);
     TaskWatcher watcher = new TaskWatcher(transport, "UPLOAD")) {

    watcher.onProgress(p -> System.out.println("upload at " + p + "%"));
    watcher.awaitStart(Duration.ofSeconds(30));
    System.out.println(watcher.awaitCompletion());
}
Mind the startup order: with this model, the client connects to a task that has already opened the port. If the client starts before the task server is reachable, the connection fails with TaskUnreachableException (handle it at application level: retry or report the failure). Events emitted before the connection are not replayed.

Delivery error handling

If publishing an event fails (for example because an observer throws an exception), the task does not fail by default: the error is logged and the work continues. You can choose the policy:

Error policies
TaskTelemetry.builder()
    .errorHandler(TaskTelemetryErrorHandler.ignore())   // silent
    // .errorHandler(TaskTelemetryErrorHandler.logging())  // default: log
    // .errorHandler(TaskTelemetryErrorHandler.rethrow())  // rethrow (the task fails)
    .build();

Event types and transported data

TypeWhenUseful data
STARTEDWhen the reporter opens—
PROGRESSprogress(...)progress (0–100) + message
INFOinfo(...)message
WARNINGwarning(...)message
HEARTBEATAutomatic, during silence—
COMPLETEDcompleted(...)message
FAILEDfailed(...)message (Throwable.toString())

Every event carries only scalar data plus message and progress: they travel identically in the same JVM and between processes. If the observer needs structured data, put it in message (for example a JSON string interpreted by the consumer).

What it does NOT do (by design)

To stay simple and predictable, the library does not:

  • execute or schedule tasks (it only observes them);
  • guarantee delivery or keep history (it is best-effort and live);
  • allow commands from the consumer to the task (communication is outbound only);
  • act as a message broker or reactive library.

That is all. Open a reporter, emit events, observe them. Build useful things, preferably without inventing another scheduler by accident.
